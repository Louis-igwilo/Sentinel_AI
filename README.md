# Sentinel AI – Insider Threat Detection System

## Quick Start (Local)

```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Make sure your model and data files are in place:
#    isolation_forest_model.joblib  (root directory)
#    data/FINAL_insider_results.csv

# 3. Run
python app.py
# Open http://localhost:5000
```

## Deploying to Render

1. Push all files to a GitHub repository.
2. Go to https://render.com → New → Web Service → Connect your repo.
3. Set **Start Command**: `gunicorn app:app --workers 1 --threads 2 --timeout 60`
4. Set environment variable: `SENTINEL_SECRET` = any long random string.
5. Upload `isolation_forest_model.joblib` and `data/FINAL_insider_results.csv`  
   via Render's Disk (configured in `render.yaml`).

## Real-World Scaling and Production Usage

This dashboard is a demo of the full detection workflow and it is designed so the same features can be scaled into a real enterprise deployment.

- **Log collection and preprocessing** would be implemented as a separate pipeline in production. Real organizations ingest logs from multiple sources (SIEM, Windows Event Logs, proxies, DLP, VPN, email gateways) and normalize them into the feature schema the model expects.
- **Feature engineering** is the critical production component. In this prototype, synthetic and dataset-based feature vectors are generated by `extract_features(...)`. In a real environment, the same model input shape can be produced by a data pipeline that maps raw event fields to `FEATURE_COLS`.
- **Model inference** is already isolated in `risk_engine(...)`. That component can be turned into an API/microservice or batch scoring service and called from any enterprise platform.
- **SHAP explainability** is integrated in the app using the real model. This is exactly the kind of XAI output needed for analysts to understand why a user was flagged.
- **Dashboard features** such as live threat queue, alert details, investigation cases, account lock/unlock actions, and log upload are all presentation-layer capabilities. They can scale horizontally by separating UI & API services and using shared backend storage or message buses.
- **Admin and analyst workflows** are already modeled in the UI. In production, account lifecycle, role-based access, and audit trails would be backed by a real identity store and database.

> In short: the current project is a small-scale demo implementation of the end-to-end workflow. The dashboard can support the same feature set in production, while the heavy lifting is scaled in the preprocessing and data ingestion layers that feed the model.
